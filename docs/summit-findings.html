Summit Findings vs Our Skills Analysis

Direct Contradictions

Places where the summit findings directly undermine claims or assumptions in our original analysis.

1. Instruction Budget Ignored

Dexter Horthy (HumanLayer) — "Everything We Got Wrong About RPI"

What Our Analysis Said

Praised repos with more skills/agents as higher-value. superpowers-ruby (28 skills), gstack (28 commands), and agency-agents (150+ agents) scored highest on "Breadth of Coverage" and "Workflow Orchestration." More = better was the implicit assumption.

What the Summit Found

Frontier LLMs reliably follow ~150-200 instructions. Dex's original 85-instruction prompt was already too many, causing critical workflow steps to be skipped ~50% of the time. He split into prompts with <40 instructions each. "Mind your instruction budget" was a core takeaway.

Impact: The repos we rated highest for breadth may actually perform worst in practice. Loading 28 interdependent skills or 150+ agent personas into context would blow the instruction budget and degrade the model's ability to follow any of them. Our analysis lacked a "context efficiency" dimension entirely.

2. Plans Aren't Leverage

Dexter Horthy (HumanLayer) — on plan vs code review

What Our Analysis Said

Positioned plan generation as a major value-add across all workflow repos. Praised gstack's 7-phase pipeline, superpowers-ruby's brainstorm-plan-execute-review, and NEXUS's 7-phase pipeline. "Structured Development Workflows" was highlighted as a key overlap theme.

What the Summit Found

A 1,000-line plan produces ~1,000 lines of code (within 10%). Plans contain surprises — the implementation will differ. Reading the plan is the same amount of work as reading the code, so you end up doing double work. "Don't read the plans. Please read the code."

Impact: The real leverage is in shorter alignment artifacts: 200-line design discussions (where are we going?) and 2-page structure outlines (how do we get there? — like C header files). None of the 6 repos produce this kind of lightweight alignment doc. They all generate full implementation plans, which Dex now considers low-leverage.

3. Automating Decisions Is the Trap

Dexter Horthy, Jake (Netflix) — "Do not outsource the thinking"

What Our Analysis Said

Treated automation depth as a positive. Praised gstack's /autoplan (automated CEO-Design-Eng pipeline), agency-agents' NEXUS full automation with quality gates, and multi-agent orchestration broadly. Higher automation = higher score.

What the Summit Found

"Do not outsource the thinking. You the engineer are an important part of this process." The design discussion format forces the agent to brain-dump all assumptions, decisions, and open questions so the human can do "brain surgery" on the agent's understanding before any code is written.

Impact: Workflows that automate decision-making (gstack's /autoplan, NEXUS full mode) are optimizing for the wrong thing. The highest-leverage moment is when the engineer corrects the agent's wrong assumptions on a 200-line doc — before 2,000 lines of code get written. The analysis should have evaluated "how much does this workflow keep the human in the decision loop?" not "how much does it automate?"

4. Research Must Be Objective

Dexter Horthy — on research contamination

What Our Analysis Said

Did not distinguish between objective and opinionated research. Evaluated research capabilities by whether the repo had them, not by how they were structured. No repo was flagged for mixing implementation intent with codebase research.

What the Summit Found

"Research == Compression of Truth." When you tell the model what you're building during research, it injects opinions instead of facts. Fix: deterministically separate context windows — one to generate questions from the ticket, a fresh one with no knowledge of the ticket to produce research. "Helpful assistants are trained to confirm our biases."

Impact: Any research skill that lets the model see the ticket/feature description while researching the codebase will produce contaminated, opinion-laden research. This is a structural design flaw that our analysis didn't evaluate. The fix is architectural: separate context windows for question generation and research execution.

5. Magic Words Are a Design Flaw

Dexter Horthy — on onboarding failures

What Our Analysis Said

Praised superpowers-ruby's "mandatory skill invocation" pattern and gstack's specific slash commands as features. Treated knowing which of 28 commands to use as acceptable complexity. Called it "process gates, not suggestions."

What the Summit Found

"If you built a tool that requires hours and hours of training and reps to get good results from, go fix the tool." Dex found himself in workshops saying "guys, don't forget to say the magic words" and called it "quite frankly, embarrassing." The fix: use deterministic control flow, not prompt-based instructions.

Impact: Repos that require users to know exactly which of 28 commands to invoke exhibit the "magic words" anti-pattern. A well-designed system should route to the right behavior deterministically based on input classification, not rely on the user (or the model) knowing the right incantation. Our analysis should have penalized complex invocation requirements.

6. Vertical Plans Over Horizontal Plans

Dexter Horthy — on plan structure

What Our Analysis Said

Evaluated planning capabilities without distinguishing plan structure. Did not assess whether repos produce vertical (end-to-end slices with checkpoints) or horizontal (layer-by-layer) plans.

What the Summit Found

Models default to horizontal plans: all database, then all services, then all API, then all frontend. This produces 1,200+ lines with nothing testable until the end. Vertical plans (mock endpoint → wire frontend → add services → database → integrate) have checkpoints where you can verify correctness before proceeding. "Despite every single model and trying to prompt this out, we cannot get models to stop writing horizontal plans."

Impact: Planning skills that don't explicitly enforce vertical slicing will produce horizontal plans by default. This is a known, persistent model behavior that requires explicit countermeasures. None of the repos address this. Any planning skill should include vertical plan structure as a hard constraint.

Strong Tensions

Not direct contradictions, but significant friction between our analysis assumptions and summit findings.

Agent Swarms Skepticism

Dexter Horthy — on quality vs speed

What Our Analysis Said

Gave agency-agents 10/10 for multi-agent orchestration. Praised NEXUS's 150+ agents across 7 phases. Treated multi-agent coordination as a uniformly positive capability.

What the Summit Found

"I'm a little mid on agent swarms and the whole gas town thing because you still need to be able to ensure quality. Going 10x faster doesn't matter if you're going to throw it all away in 6 months." Target 2-3x with near-human quality, not 10x with slop.

Impact: More agents is not inherently better. The analysis should have evaluated whether multi-agent setups include quality verification at each handoff, not just whether they have many agents. 150 agents with no quality guarantees may be worse than 5 well-verified ones.

Adoption Readiness

Scott Breitenother (Kilo Code) — 25T token analysis

What Our Analysis Said

Evaluated repos on capability and breadth without considering whether users are ready for that level of complexity. Scored "Ease of Adoption" but didn't connect it to the trust ladder.

What the Summit Found

From 25T+ tokens across 1.5M developers: adoption follows a trust ladder (autocomplete → chat → single agents → orchestration). "If autocomplete fails, agents never get a chance." 49% of pro devs don't use AI daily. Developers climb, they don't jump.

Impact: Most repos target the top of the trust ladder (orchestration). If a developer hasn't built trust at the agent level, giving them a 150-agent NEXUS framework is counterproductive. The analysis should have evaluated "what rung of the trust ladder does this repo assume?"

Missing Dimensions

Critical evaluation axes the summit revealed that our analysis completely lacked.

Context Efficiency

Dexter Horthy, Ankit Mathur (Databricks)

Our analysis never evaluated how much context window each repo consumes. The Databricks team noted "you have too many MCPs, the whole context window is full of instructions about how to use tools you don't care about." Dex showed that beyond ~40% context utilization, results degrade. The analysis should have included: instructions per skill, total context footprint, and whether the repo uses static artifacts (resumable across sessions) vs in-context state (lost on compaction).

New evaluation question: "If I load this repo's skills into my context window, how much budget is left for my actual task?"

Learning & Memory Architecture

Erin Ahmed (Cleric), Faye Zhang (Pinterest)

Erin Ahmed argued that "agent capabilities are commoditized — the next horizon of differentiation is learning." Agents that don't persist corrections, compound knowledge, or absorb context continuously won't survive. Three lessons: make corrections easy, reward corrections with visible improvement, absorb context continuously. Faye Zhang identified "memory collapse" as a key agent failure mode and proposed 3-tier memory (hot, domain task, cold storage). Our analysis noted agency-agents' memory sections don't persist as a minor weakness. It should have been flagged as a fundamental architectural gap across all 6 repos.

New evaluation question: "Does this repo help the agent learn from past mistakes, or does every session start from zero?"

Security as a First-Class Concern

Milan Williams (Semgrep), Ankit Mathur (Databricks)

Milan Williams outlined a practical security framework: (1) scoped credentials before starting, (2) audit logging via hooks during sessions, (3) automated code scanning before shipping. Databricks highlighted MCP authentication as a security nightmare — plain text tokens, locally stored, no rotation. Our analysis evaluated "Safety & Guardrails" but focused on preventing the agent from breaking things (gstack's /freeze, counselors' read-only). We didn't evaluate credential scoping, audit trails, or vulnerability scanning of generated code.

New evaluation question: "Does this repo help me control what the agent can access, log what it did, and verify the output is secure?"

Agentic Search vs Vector Search

Jessica Wang (Braintrust)

Braintrust's eval showed agentic search (grep, find, read — what Claude Code uses) matched or beat vector search while using 3x fewer tokens and costing 2.8x less per task. Vector search returns chunks without "connective tissue." Our analysis didn't evaluate how repos approach codebase search. Repos that rely on RAG pipelines for context may be spending more for worse results than those using Claude Code's native tools.

New evaluation question: "Does this repo use native code exploration or add a RAG layer that may not improve outcomes?"

What Held Up

Analysis findings that the summit validated or reinforced.

✓

Structured Process Beats Ad-Hoc Prompting

Every summit speaker reinforced that having a structured workflow matters. Dex just argues the structure should be lighter (design discussion + outline) rather than heavier (full implementation plan). The principle is validated; the implementation needs refinement.

✓

Code Review Is Universally Valued

Dex ("please read the code"), Databricks ("commenting on code reviews I feel like the author often also sees the code for the first time"), and Semgrep (scan before shipping) all reinforce that 5 of 6 repos having dedicated review is the right instinct. If anything, the analysis underweighted the importance of reviewing actual code vs reviewing plans.

✓

"Build for Yourself, Steal the Patterns"

Our takeaway #6 was "study the patterns, then write skills that encode YOUR conventions." Dex's core message: "There is no magic prompt." Mihail Eric (Stanford): "This is something you should iterate on to find what works for you." The summit unanimously validates that custom workflows beat drop-in solutions.

✓

"Don't Adopt Everything at Once"

Our takeaway #5 is directly supported by the instruction budget finding. If the model can only follow ~200 instructions, loading all 28 skills from any repo is self-defeating. Start small. The summit provides the scientific backing for what our analysis recommended on instinct.

✓

Evidence-Based Verification

Our "trust but verify" theme (superpowers-ruby's verification-before-completion, gstack's QA, agency-agents' Reality Checker) maps directly to Dex's "please read the code" and Databricks' "verification is king with LLMs." The analysis got the principle right. The summit says apply it to code, not plans.

✓

Codebase-First Philosophy

rails-conventions' "mandatory inspection before proposing changes" aligns with Dex's objective research principle. Scan what exists, don't impose opinions. This is the right instinct — the summit adds that the scan should happen in a context window that doesn't know what you're about to build.

Revised Recommendations

How the original analysis takeaways should change in light of summit findings.

1 Evaluate context efficiency, not breadth

Before adopting any repo's skills, count the instructions. If a single skill has 40+ instructions, it's already risky. If you're loading multiple skills, you need their combined instruction count well under 200. Prefer repos that produce static artifacts (files you can resume from) over those that rely on in-context state.

  • Old advice: "Workflow structure is the highest-leverage investment"
  • New advice: "Lightweight workflow structure with minimal instruction footprint is the highest-leverage investment"

2 Replace plan review with design alignment

Instead of generating and reviewing 1,000-line implementation plans, generate a ~200-line design discussion (where are we going, what patterns to follow, resolved decisions, open questions) and a ~2-page structure outline (phases, signatures, types). Review those. Then let the agent write the plan for its own reference and go straight to code review.

  • Old advice: "Plan before you code, review the plan"
  • New advice: "Align before you code (design + outline), review the code"

3 Keep the human in the decision loop

Evaluate workflows by how much they force the agent to surface its assumptions and ask questions, not by how much they automate. The best moment in any workflow is when the agent shows you what it found and what it plans to do, and you correct its wrong assumptions on a short doc rather than in 2,000 lines of code.

  • Old advice: "Higher automation = higher value"
  • New advice: "Higher alignment quality = higher value; automation serves alignment, not the reverse"

4 Use control flow, not prompts, for routing

If your workflow has multiple phases, don't encode all phases in one prompt and hope the model follows them. Use actual if-statements and input classification to route to smaller, focused prompts. "Don't use prompts for control flow if you can use control flow for control flow."

  • Old advice: "Process gates in prompts"
  • New advice: "Process gates in code; prompts handle one focused task"

5 Separate research from implementation intent

When researching a codebase, use a context window that doesn't know what you're about to build. Generate questions from the ticket in one context, then run objective research in a fresh context. This prevents the model from confirming your biases.

  • Old advice: Not addressed
  • New advice: "Research context windows should never see the ticket. Research == compression of truth, not opinion."

6 Enforce vertical plan structure

If you use planning at all, explicitly constrain plans to vertical slices (end-to-end features with testable checkpoints) rather than horizontal layers (all DB, then all services, etc). Models default to horizontal. This requires explicit countermeasures in the planning prompt.

  • Old advice: Not addressed
  • New advice: "Every planning skill should enforce vertical slicing with intermediate verification checkpoints"

7 Add security controls to your agent setup

Before evaluating what a repo can do, evaluate what it controls. Scoped credentials, audit logging, and code scanning are non-negotiable for production work. "If you wouldn't give it to an intern, why'd you give it to your agent?"

  • Old advice: "Safety & Guardrails" evaluated as preventing agent mistakes
  • New advice: "Security is three layers: scoped access before, audit trails during, vulnerability scanning after"
