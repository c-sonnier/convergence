Key Overlaps & Themes

Where these repos converge reveals what matters most in Claude Code configuration.

Structured Development Workflows

superpowers-ruby gstack agency-agents compound-engineering

All four implement phased pipelines where you don't jump straight to coding. superpowers-ruby gates on brainstorming and planning. gstack has a 7-phase sprint. NEXUS runs 7 phases from discovery to operations. compound-engineering adds a knowledge compounding loop. The consensus: structured process beats ad-hoc prompting.

Code Review as First-Class Skill

superpowers-ruby gstack agency-agents counselors palkan/skills compound-engineering

6 of 7 repos have dedicated review capabilities. superpowers-ruby uses two-stage review (spec compliance + code quality). gstack provides staff-engineer-level PR review. counselors gets independent perspectives from multiple models on the same diff. palkan detects layer violations. compound-engineering has multiple specialized reviewers (Rails, TypeScript, Python, security, architecture). This is clearly essential infrastructure.

37signals / DHH Rails Philosophy

superpowers-ruby palkan/skills rails-conventions

Three repos encode 37signals/Basecamp patterns: thin controllers, rich models, vanilla Rails, Hotwire-first. superpowers-ruby has the deepest treatment (dedicated skill with DHH/Jorge Manrubia references). rails-conventions makes it a core philosophy. palkan acknowledges it while proposing a complementary layered approach.

Multi-Agent Orchestration

superpowers-ruby gstack agency-agents counselors compound-engineering

Five repos tackle multi-agent coordination differently. superpowers-ruby isolates subagent context carefully. gstack runs parallel review agents. agency-agents' NEXUS is the most ambitious (150+ agents across 7 phases). counselors takes a unique approach: same prompt, different models, compare results. compound-engineering dispatches 50+ specialized agents. Different strategies, same insight: one agent isn't enough.

Evidence-Based Verification

superpowers-ruby gstack agency-agents compound-engineering

All four enforce "prove it, don't claim it." superpowers-ruby has a dedicated verification-before-completion skill (no claims without fresh evidence). gstack's QA generates regression tests for every fix. agency-agents' Reality Checker defaults to "NEEDS WORK" until proven otherwise. compound-engineering's reviewers default to "NEEDS WORK" as well. Trust but verify is table stakes.

Safety Guardrails

gstack counselors

gstack has the most sophisticated safety system: /careful warns before destructive commands, /freeze blocks edits outside a directory, /guard combines both. counselors enforces read-only by default across all tool adapters. Both treat safety as a non-negotiable feature, not an afterthought.

Multi-Tool Portability

superpowers-ruby agency-agents counselors

Not everyone uses Claude Code. superpowers-ruby works across Claude, Cursor, Codex, and OpenCode. agency-agents converts agents to 10+ tool formats (Cursor, Copilot, Windsurf, Aider, Kimi, etc.). counselors orchestrates 5 different AI CLIs. The trend: don't lock into one tool.

Testing Discipline

superpowers-ruby gstack palkan/skills rails-conventions compound-engineering

superpowers-ruby enforces strict TDD (red-green-refactor always, "Iron Law"). gstack runs QA with tiered severity and auto-generates regression tests. palkan uses the "specification test" as a design tool to detect layer violations. rails-conventions mandates minitest + fixtures. compound-engineering adds spec-flow analysis for completeness gaps. All agree: tests aren't optional.

Rails Architecture Patterns

palkan/skills rails-conventions superpowers-ruby

palkan goes deepest with 4-layer architecture, callback scoring, and god object detection. rails-conventions provides quantified code quality gates (class < 200 lines, method < 10 lines). superpowers-ruby embeds Sandi Metz rules (classes < 100 LOC, methods < 5 LOC). Different thresholds, same goal: keep code small and focused.

Security Scanning

superpowers-ruby gstack counselors rails-conventions compound-engineering

superpowers-ruby integrates Brakeman for Rails-specific static analysis. gstack's /cso does OWASP Top 10 + STRIDE threat modeling. counselors has a security preset for multi-model review. rails-conventions provides a manual checklist. compound-engineering has a dedicated security sentinel agent. Multiple approaches, universal concern.

Knowledge Compounding

compound-engineering

Only one repo addresses persistent learning across sessions. compound-engineering's ce:compound skill captures learnings after work; its learnings-researcher agent surfaces them before new work. No other repo attempts this, despite it being identified as a critical gap by the Coding Agents Summit.

Deep Dives

Expand each repo for detailed findings, unique features, and what's worth borrowing.

superpowers-ruby — The Disciplined Rails Workflow

What Makes It Unique

  • Mandatory skill invocation — Skills are process gates, not suggestions. The "using-superpowers" skill includes red-flag rationalizations agents make when trying to skip steps
  • Context isolation for subagents — Each subagent gets only the exact context needed, preventing cognitive pollution
  • Two-stage review — Separates "did we build the right thing?" (spec compliance) from "did we build it right?" (code quality)
  • TDD applied to documentation — The writing-skills skill tests docs like code: watch agents violate without it, write the doc, verify compliance
  • Zero-dependency brainstorm server — Node.js WebSocket server built from scratch (RFC 6455), auto-exits after 30min idle

Key Skills Worth Studying

  • brainstorming — Hard-gate before creative work, Socratic dialogue, spec review loops (max 3 iterations)
  • subagent-driven-development — Fresh subagent per task + two-stage review
  • systematic-debugging — "No fixes without root cause first" with 4-phase investigation
  • verification-before-completion — Gate function: identify proof command → run → read output → verify → claim WITH evidence
  • 6 Hotwire Club skills — Deepest Turbo/Stimulus coverage of any repo

Weaknesses

  • 28 interdependent skills = large surface area for conflicts
  • Ruby/Rails-only; no guidance for other languages
  • Mandatory skill invocation relies on instruction compliance, not runtime enforcement
  • Limited non-TDD guidance (prototyping workflows underdeveloped)
gstack — The Software Factory

What Makes It Unique

  • Persistent browser daemon — Chromium stays alive between commands (~100ms latency). Cold start ~3s, then near-instant. Can launch real Chrome for co-presence
  • "Boil the Lake" philosophy — When AI makes completeness cheap, always do the complete thing. Tests, edge cases, error paths, docs — all cheap now
  • Cross-model analysis — When both /review (Claude) and /codex (OpenAI) run on same diff, generates overlap and unique findings
  • Safety hooks — /careful warns on destructive commands, /freeze locks edits to a directory, /guard combines both. Mid-session activation
  • Contributor mode — Self-improving: rates experience 0-10, auto-files issue reports for anything below 10
  • Template system — Skills maintained as .tmpl files, generated into SKILL.md via TypeScript build pipeline

Key Commands Worth Studying

  • /office-hours — YC-style diagnostic with 6 forcing questions before any building
  • /autoplan — Automated CEO → Design → Eng review pipeline, surfaces only taste decisions
  • /qa — QA lead mode with 3 tiers (Quick/Standard/Exhaustive), before/after health scores, auto regression tests
  • /cso — OWASP + STRIDE threat modeling, secrets archaeology, dependency supply chain audit
  • /retro — Weekly engineering retrospective from commit history with per-person breakdowns

Weaknesses

  • Assumes web apps + Git + GitHub (limited for mobile, monorepos, non-GitHub)
  • No visual regression testing (Percy/Chromatic equivalent)
  • Steep learning curve; philosophy-heavy onboarding
  • E2E test framework is complex for contributors
agency-agents — The Virtual Organization

What Makes It Unique

  • 150+ specialized personas — Not generic prompts. Each has distinct voice, methodology, and mental model. Reddit Community Builder understands subreddit authenticity rules
  • 14 divisions — Engineering, Marketing, Design, Testing, Sales, Support, Spatial Computing, Game Dev, Academic, Strategy, Product, Project Management, Paid Media, Specialized
  • NEXUS orchestration — 3 deployment modes (Full/Sprint/Micro), 7 phases, quality gates, handoff templates, max 3 retries before escalation
  • Reality Checker agent — Defaults to "NEEDS WORK" until overwhelming evidence proves readiness. Demands screenshots, test results, actual verification
  • 10+ tool format conversions — Automated scripts convert all agents to Cursor, Copilot, Windsurf, Aider, Kimi, OpenCode, Gemini, and more
  • China market depth — 15+ agents for Xiaohongshu, Douyin, WeChat, Baidu, Zhihu, Bilibili with deep cultural context

Key Agents Worth Studying

  • Reality Checker — QA pattern that prevents fantasy approvals
  • Workflow Architect — Maps every system path before implementation, including failure modes
  • Discovery Coach — Deep SPIN/Gap Selling/Sandler frameworks for sales
  • MCP Builder — Agent-friendly tool interface design for Model Context Protocol
  • Agents Orchestrator — Meta-agent that coordinates other agents in NEXUS pipelines

Weaknesses

  • Agents are personality overlays, not autonomous executors — still requires human/meta-agent orchestration
  • No persistent memory across sessions despite "Memory" sections in agent definitions
  • Handoff templates exist but no enforcement mechanism
  • 150+ agents is overwhelming; no clear "start here" path
counselors — The Council of Advisors

What Makes It Unique

  • Multi-model same-prompt dispatch — Not task parallelization; gets independent perspectives from Claude, Codex, Gemini, and Amp on the same question
  • Read-only by default — Three enforcement tiers: enforced, bestEffort, none. CLI policy filters tools by strictness
  • Multi-round convergence — Loop mode augments round 2+ prompts with prior outputs. Auto-stops when new findings drop below 30% of prior round
  • Environment security — Allowlist + denylist for env vars. Blocks NODE_OPTIONS, LD_PRELOAD, DYLD_INSERT_LIBRARIES to prevent sandbox bypasses
  • No infrastructure — Just calls local CLI binaries. No containers, git worktrees, or complex setup

Key Presets Worth Studying

  • bughunt — Logic errors, boundary failures, concurrency bugs, resource leaks with severity/confidence scoring
  • security — Injection, auth, access control, XSS, deserialization, crypto weaknesses
  • hotspots — O(n²)+ patterns, N+1 queries, unbounded traversal with Big-O before/after analysis
  • invariants — State synchronization, impossible states, boolean explosion detection
  • contracts — API contract drift across handlers, clients, validators, tests

Weaknesses

  • Synthesis is structural (table + headings), not LLM-synthesized
  • Only 6 presets; many domains unaddressed
  • No built-in consensus/voting logic
  • Cost tracking only works for Amp
  • Models hardcoded in adapters; custom models need adapter subclassing
palkan/skills — The Architecture Specialist

What Makes It Unique

  • Callback scoring system (1-5) — Removes subjectivity from callback debates: 5=Transformer (keep), 4=Maintainer (keep), 3=Timestamp (acceptable), 2=Background trigger (consider), 1=Operation (extract)
  • Specification test as design tool — Uses test structure (describe/context blocks) to detect layer violations. If a model test needs contexts about "when notification fails," that code doesn't belong there
  • Churn × complexity heuristics — Finds god objects algorithmically. High churn + high complexity = candidate. Avoids false positives
  • 4-layer architecture — Presentation → Application → Domain → Infrastructure with unidirectional data flow
  • "Gradual layerification" — Phased roadmaps with escape hatches ("stop here if..."), not big-bang rewrites

Key Commands Worth Studying

  • /layers:analyze — Full codebase architecture analysis: layer violations, god objects, callback scoring, missing abstractions
  • /layers:analyze:callbacks — Deep callback audit with 1-5 scoring for every callback in codebase
  • /layers:analyze:gods — Churn × complexity detection with responsibility cluster mapping
  • /layers:spec-test — Generate test skeleton, annotate each context as belongs/borderline/violation
  • /layers:gradual — Incremental adoption planning for specific goals (authorization, callbacks, god objects)

Weaknesses

  • Rails-only; no guidance for non-Rails Ruby or other languages
  • No performance considerations (N+1, eager loading, caching)
  • Limited testing strategy beyond specification test
  • No Hotwire/API-first/async patterns
  • Minimal example refactoring code
rails-conventions — The Pragmatic Reference

What Makes It Unique

  • Codebase-first scan — Mandatory inspection before proposing changes. Respects existing patterns (if app uses RSpec, don't push Minitest)
  • Smart backend detection — Checks config.active_job.queue_adapter → Gemfile → runtime config. Routes to good_job or solid_queue guide conditionally
  • Quantified code quality gates — Class < 200 lines, < 15 public methods; Method < 10 preferred, < 20 hard limit; grep patterns to detect violations
  • Reference routing — SKILL.md maps task types to specific references so you don't read all 15 docs for every task
  • Fail-fast philosophy — Avoid defensive patterns (respond_to?, try, broad rescue). Prefer explicit contracts (find_by!, fetch)

Key References Worth Studying

  • 03 Models & Data — State records vs booleans, write-time optimization, query safety
  • 06 Hotwire Turbo Stimulus — Progressive enhancement, morphing, model broadcasts
  • 07a/07b Background Jobs — Split good_job vs solid_queue with operational guidance
  • 09 Security Checklist — Specific patterns for XSS, CSRF, SSRF, IDOR, injection
  • 12 37signals Profile — Vanilla Rails philosophy, rule of three extraction, fix root causes
  • 13 Code Quality Gates — Quantified thresholds with detection patterns

Weaknesses

  • Pure reference; no automated linting, checking, or CLI tools
  • No agents, hooks, or interactive tooling
  • Frontend underspecified (only Hotwire; no React/Vue/GraphQL)
  • No monitoring/observability guidance
  • No deployment or infrastructure patterns
compound-engineering — The Knowledge Compounder

What Makes It Unique

  • Knowledge compounding loop — ce:compound captures learnings after work; learnings-researcher surfaces them before new work. The only repo to implement persistent learning across sessions
  • 42 skills across 4 platforms — Claude Code, Cursor, Windsurf, VS Code transpiler. Most comprehensive multi-platform support
  • 50+ specialized agents — Reviewers for Rails, Python, TypeScript, frontend race conditions, architecture, security, data integrity, deployment verification
  • Spec-flow analysis — Analyzes specifications for user flow completeness, gap identification, and edge case discovery
  • Design-to-implementation sync — Figma design comparison agents for visual fidelity verification
  • Destructive command hooks — Warns before rm -rf, DROP TABLE, git push --force, git reset --hard

Key Patterns Worth Studying

  • ce:compound — 530-line skill with 3 modes, YAML frontmatter for searchable learnings with tags, module, severity
  • learnings-researcher — Agent that searches past learnings by frontmatter metadata and body text before new work
  • Specialized review agents — Different reviewers for different domains (Rails conventions, TypeScript patterns, frontend race conditions, security)
  • Bug reproduction validator — Systematically reproduces bug reports before attempting fixes

Weaknesses

  • 42 skills at 100-530 lines each blows the instruction budget (summit finding: <35 per skill)
  • 50+ agents = overwhelming surface area, no clear entry point
  • Heavy plans and multi-agent pipelines contradict summit findings on plan leverage
  • Multi-platform transpiler adds complexity without clear benefit (each platform has its own conventions)
  • Most patterns already covered by other repos — the unique contribution is knowledge compounding

Key Takeaways

What this means for building YOUR setup.

1. These repos solve different problems

They look similar on the surface but serve distinct purposes. superpowers-ruby, palkan/skills, and rails-conventions are Rails-specific knowledge bases. gstack is a general-purpose workflow engine. agency-agents is an agent persona library. counselors is a multi-model orchestration tool. compound-engineering is a knowledge compounding system. A custom setup would cherry-pick from multiple categories.

2. Lightweight workflow structure is the highest-leverage investment

The repos that deliver the most value enforce a development pipeline: think before you build, align before you code, verify before you ship. But summit findings show that heavyweight plans (1,000+ lines) aren't leverage — they're the same work as reading the code. The real leverage is in short alignment artifacts: ~200-line design discussions and 2-page structure outlines. Review those, then review the code. And mind the instruction budget: LLMs reliably follow ~150-200 instructions, so each skill must be lean.

3. Code review is universally valued

6 of 7 repos invest heavily in review capabilities. The approaches vary (spec compliance, layer violations, multi-model comparison, staff-engineer mode, specialized domain reviewers) but the message is clear: agents reviewing agents' work catches things single-pass doesn't. Build review into your workflow early.

4. Rails-specific: combine palkan + rails-conventions + cherry-pick from superpowers

For Rails work specifically: palkan's callback scoring and god object detection are unique and actionable. rails-conventions' codebase-first philosophy and backend detection are practical. superpowers-ruby's Hotwire skills and TDD enforcement fill gaps the others miss. Together they cover more than any single repo.

5. Don't adopt everything at once

superpowers-ruby has 28 skills. gstack has 28 commands. agency-agents has 150+ personas. compound-engineering has 42 skills and 50+ agents. Adopting all of any of these would be overwhelming. Start with: (1) a workflow skill matching your process, (2) a review skill, (3) domain-specific reference material for your stack. Add more only when you feel a gap.

6. Build for yourself, steal the patterns

The most useful thing in these repos isn't the content — it's the patterns. How superpowers-ruby structures a skill. How gstack uses hooks for safety. How palkan scores callbacks. How counselors enforces read-only. How agency-agents formats agent personas. Study the patterns, then write skills that encode YOUR conventions, YOUR workflow, YOUR preferences.

7. Summit caveat: context efficiency changes the rankings NEW

The Coding Agents Summit revealed that LLMs reliably follow ~150-200 instructions max. This inverts some of our rankings: repos with the most skills/agents may perform worst in practice because they blow the instruction budget. The repos we scored lowest on breadth (rails-conventions, palkan/skills) may actually be the most effective because they're lean.
